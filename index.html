0开篇：这份材料怎么用 + 比喻总表

  • 是 PolarDB（共享存储，一写多读），还是 PolarDB-X（数据真分片的分布式）？

1第一章 · PolarDB 家族全景

1.1 产品定位

目标：在兼容生态（MySQL/PG）前提下，提供比传统主备更弹性的计算、更快的只读扩展、云上高可用与备份。

1.2 家族对照

产品数据放置写扩展读扩展典型替换对象
PolarDB MySQL共享存储升配主节点加只读节点RDS MySQL / 自建 MySQL
PolarDB PostgreSQL共享存储升配主节点加只读节点RDS PG / 自建 PG
PolarDB-X多分片 DN加分片/扩 DNCN/DN 扩展分库分表中间件、超大单库
选型口诀
  单库还能扛、主要缺只读与弹性 → PolarDB 共享存储版
  单主写打满、百 TB 级、要水平扩写 → PolarDB-X
  不确定 → 先压测写 TPS / 数据量 / 跨片事务比例再定

2第二章 · 传统架构痛点与 PolarDB 解题

2.1 传统「柜台+私人账本」

  • 主备各有完整磁盘副本；备库靠日志追赶 → 复制延迟；
  • 加只读 ≈ 再搬一套数据，慢；
  • CPU 与磁盘绑死，弹性差。

2.2 PolarDB「多柜台+共享金库」

  • 计算节点可横竖扩展；
  • 权威数据在共享存储，只读节点轻量拉起；
  • 失败切换更多是「换柜台」，金库还在。
传统主备：主账本 ──抄小票──▶ 备账本（可能旧）
PolarDB：  主柜台 ──读写──▶ 共享金库 ◀──只读── 只读柜台

3第三章 · 共享存储版核心架构（深度）

3.1 逻辑分层

  1. 接入层：集群地址/代理，做读写分离、屏蔽切换；
  2. 计算层：SQL 解析、优化器、事务锁、缓冲池；
  3. 存储层：分布式共享存储，负责页与日志持久化、多副本可靠。

3.2 写路径（原理）

UPDATE
  → 代理（可选）→ 主计算节点
  → 加行锁/表锁（按隔离级别）
  → 修改缓冲中的数据页 + 产生重做日志
  → 日志/页持久化进入共享存储（满足落定语义）
  → 返回客户端成功
  → 只读节点通过共享存储与同步协议获得新可见版本

3.3 读路径

SELECT
  → 代理按策略：主 or 只读
  → 计算节点缓冲命中？命中则还
  → 否则向共享存储拉页
  → 返回结果集

3.4 深度要点

  • 一写多读：常态仅一个写者，避免多主冲突；
  • 计算无状态化趋势：节点可替换，状态尽量在存储；
  • 本地盘角色：多为临时/日志缓冲/缓存，不是业务权威副本（以产品实际为准）。
记住：应用看见的「库」，是「计算入口」；数据耐造靠「共享金库」。

4第四章 · 复制、一致性级别与粘主

4.1 现场高频：写完读旧

支付成功页若走只读地址，可能读到切换前快照。处理分级：

接口类型建议
支付结果/余额/冲正前核对粘主（读主节点）
订单列表、内容浏览可读只读
报表分析只读 + 接受延迟
写成功
  ├─ 强一致需求 → 读主 / 会话粘滞主
  ├─ 可延迟 → 读只读（监控延迟水位）
  └─ 延迟超阈 → 摘流或降级提示

4.2 原理

只读可见性依赖同步协议与水位。运维要会看：只读延迟、是否可接流量、切换后连接是否漂移。

5第五章 · 事务 / 锁 / MVCC / 长事务（结合 Java）

5.1 三者关系

事务 = 一捆要一起成或一起回滚的操作
锁   = 防止别人同时乱改同一行
MVCC = 用版本让读少阻塞写（通俗：多人看不同版本复印件）

5.2 Java 侧高发根因 → 库侧现象

代码味道库现象改法
@Transactional 包住远程调用长事务、锁久事务最小化
循环单条更新热点用户锁等待链批量/排队
一次删除千万行大事务、空间涨分批幂等
只读报表忘关连接版本堆积短查询+超时
捕获异常后不抛出业务以为失败，库已提交或未回滚事务边界清晰
排障顺序
  发现超时 → 查锁等待/谁占谁
          → 回追 SQL 与会话来源
          → 对到 Java 调用栈/事务边界
          → 临时杀会话（慎）+ 根治代码

6第六章 · 高可用与故障切换

主节点故障
  → 探活失败（连续）
  → 管控防脑裂决策（租约/仲裁）
  → 新主从只读/候选提升

  → 代理切流
  → 应用连接池重连 + 幂等核对半成功
  • 切换期：短拒绝或只读；
  • 最怕：应用盲重试造成双写；

7第七章 · 备份恢复与应急

8第八章 · 性能：缓冲、慢 SQL、加只读、写瓶颈

8.1 读扩展

加只读 + 代理分流。慢 SQL 会打爆只读，一样事故。

8.2 写瓶颈

写打满？
  1) 热点行/锁（业务）
  2) 低效 SQL（缺索引、大事务）
  3) 主节点 CPU/IO 升配
  4) 仍不够 → PolarDB-X 或业务拆分

9第九章 · PolarDB-X：分布式总览与组件

组件大白话职责
CN调度大厅收 SQL、计划、下推、汇总
DN分柜金库存分片、本地事务执行
元数据服务总登记处拓扑、规则、全局信息
变更组件搬家队DDL、扩缩容、同步
SQL → CN 解析
     ├─ 单片 → 下推 1 个 DN（最佳）
     ├─ 多片 → 并行 DN → CN 归并
     └─ 跨片事务 → 全局协调
与共享存储 PolarDB 最大区别：数据切开了；跨片是常态成本来源。

10第十章 · 分片键、热点、全局唯一

  • 高频交易尽量 单分片闭环（按账号/用户）；
  • 时间分片易热点；
  • 全局唯一用号段/雪花，避免各片撞号；
  • 扩容重分布=重变更，要窗口与校验。
好设计：账号ID → 固定 DN → 本地强一致事务快
坏设计：每次按手机号模糊查 → 广播所有 DN → 拖死集群

11第十一章 · 分布式事务、全局时钟、幂等

跨片转账
  CN 开全局事务
    DN_A 预提交 ──┐
    DN_B 预提交 ──┼─ 都成功？提交：否则回滚
  超时：应用用业务键查终态，禁止盲重试
  • 能单片就单片（性能与稳定性数量级差）；
  • 全局事务依赖全局次序/时钟类机制保证可见性与提交序（不同版本实现细节有别）；
  • 金融：幂等键含业务单号+动作类型。
分布式事务是税；分片设计是避税。
