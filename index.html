2第 1 章 · PolarDB 家族与版本形态

1.1 一句话定义

PolarDB 是阿里云的云原生数据库产品线：对应用尽量兼容 MySQL/PostgreSQL 生态，对内用计算与存储分离、弹性节点、共享存储（或分布式分片）来解决传统主备扩容慢、只读贵、资源绑死的问题。

1.2 家族表

产品协议兼容数据怎么放写怎么扩读怎么扩典型场景
PolarDB MySQL 版MySQL共享存储升配主节点加 RO替代大 RDS MySQL
PolarDB PostgreSQL 版PostgreSQL共享存储升配主节点加 RO企业 PG 上云
PolarDB-XMySQL 生态为主DN 分片加分片/扩 DNCN/DN 扩展超大库、高并发写

1.3 金融现场沟通清单

  1. 实例是 PolarDB 还是 PolarDB-X？
  2. 引擎版本与小版本号？
  3. 有几个 RO？代理是否开启读写分离？
  4. 备份保留几天？最近一次恢复演练何时？
  5. 是否有跨片事务/全局索引？

3第 2 章 · 为什么需要 PolarDB：传统痛点拆解

2.1 传统主备「柜台+私人账本」

  • 主、备各有完整数据文件；备靠日志追 → 复制延迟；
  • 加只读 ≈ 再搬一套盘，耗时长、成本高；
  • CPU 不够要整机升，磁盘浪费或不够都难受；
  • 切换时要确认数据追上，RTO/RPO 受复制质量影响。
传统：应用 → 主库(盘A) ──日志──▶ 备库(盘B)
问题：盘B 落后 → 读旧；加备要拷盘；计算与盘绑死

2.2 PolarDB 解题：多柜台共享金库

  • 计算节点可加减；
  • 权威数据在共享存储；
  • RO 轻量拉起，扩展读更快。
PolarDB：应用 → 代理 → 主柜台/只读柜台
                         └──── 共享金库（多副本可靠）
注意：共享存储版写仍然集中在主计算节点。写到天花板，要优化 SQL/热点，或评估 PolarDB-X。

4第 3 章 · 共享存储版架构深度（计算存储分离）

3.1 三层结构

层组件职责挂了会怎样
接入集群地址/代理路由、读写分离、屏蔽切换连不上或路由错
计算RW / RO 节点SQL、事务、锁、缓冲切换/只读摘流
存储共享分布式存储页与日志持久化、多副本写失败/只读（极端）

3.2 写路径（逐步）

1. 应用拿连接（办事窗口）打到代理/主地址
2. 主计算节点解析 SQL、鉴权、优化计划
3. 按隔离级别加锁，修改缓冲中的数据页
4. 产生重做/事务日志，持久化到共享存储（落定语义）
5. 返回客户端成功
6. RO 通过共享存储与同步机制看到新可见版本

3.3 读路径

SELECT → 代理判定走主或 RO
      → 计算节点缓冲命中？命中则返回
      → 否则从共享存储拉页 → 返回结果

3.4 架构含义（运维必懂）

  • 换计算节点 ≠ 丢数据（存储仍在），但仍要防切换中应用半成功；
  • RO 不是「另一套完整私人账本」叙事，而是共享金库上的只读计算；
  • 监控既要看节点 CPU/会话，也要看存储水位与 I/O。
一句话：多个大脑（计算），一个金库（存储）；写集中，读可加柜台。

5第 4 章 · 一写多读、复制水位与读写分离策略

4.1 现场第一案：「写完读旧」

用户支付成功，结果页走 RO，短时仍显示未支付。根因是可见性水位，不是「没写入」。

接口类型建议读哪里原因
支付结果、余额、冲正前核对必须粘主错不得
订单列表、内容浏览RO 可可接受短暂旧
报表/导出RO + 隔离别打主
写成功？
  ├─ 强一致接口 → 读主 / 会话粘滞主
  ├─ 可延迟 → 读 RO（盯延迟水位）
  └─ 水位超阈 → 摘流 RO 或提示稍后

4.2 代理策略清单

  • 是否按 SQL 类型自动分流？
  • 是否支持会话级 hint/强制主？
  • RO 全部不健康时是否回落主？（回落会加重主压）
和工行「抄写柜台」同一类问题：能读备/RO，不代表能当支付确认页。

6第 5 章 · 事务、锁、MVCC、长事务（Java 复合视角）

5.1 概念

  • 事务：一捆操作一起成功或一起回滚；
  • 锁：保护同一行不被乱改；
  • MVCC：用多版本让读尽量不堵写（看复印件）。

5.2 Java → 库现象映射

代码味道库侧现象处理
事务里调远程/睡眠长事务、锁久占缩小事务边界
循环更新同热用户锁等待链合并更新/排队
一次删改千万行大事务、空间涨、切换慢分批+幂等续跑
异常吞掉不回滚半成功、状态分裂明确回滚/提交
只读报表事务不关版本堆积、空间压短查询+超时
排障
  超时投诉 → 查锁等待/长事务
           → 找到会话与 SQL
           → 追到应用事务边界
           → 临时处置（慎杀）+ 根治代码/限流
你的优势：别人只看见锁；你能指出「事务传播/循环改账/吞异常」。

7第 6 章 · 高可用：探活、切换、脑裂防护、应用侧

主节点探活连续失败
  → 管控判定（租约/仲裁，防双主）
  → 提升 RO/候选为新主
  → 存储认新写者租约
  → 代理切流
  → 应用连接池重连
  → 用业务键核对半成功（禁盲重试）

6.1 运维检查单

  • 切换事件时间线、谁触发；
  • 切换后是否有双写窗口；
  • RO 是否追上、可否重新挂只读流量；
  • 应用错误码：是否可重试、是否幂等。
高可用不是应用免责金牌。切换瞬间最怕半成功 + 盲重试双扣。

8第 7 章 · 备份、时间点恢复、误删应急

7.1 备份在保什么

共享存储快照/数据页 + 日志链，支持按时间点恢复（以控制台与版本为准）。

定期快照 ─┐
日志保留 ─┼→ 选时间点 T → 恢复临时实例 → 校验 → 导回/订正
手工全备 ─┘

7.2 误删表标准动作

  1. 停继续破坏的操作；
  2. 确认大约时间点；
  3. 恢复到临时实例，不直接盖生产；
  4. 导出缺失数据回灌；
  5. 复盘权限与变更门禁。
指标含义
RPO最多丢多久数据
RTO多久恢复可服务

9第 8 章 · 性能与容量：RO 扩展、慢 SQL、写天花板

8.1 读扩展

加 RO + 代理分流。慢 SQL/大报表会打爆 RO，一样事故。

8.2 写扩展边界

写打满诊断顺序
  1. 热点行/锁（业务）
  2. 缺索引/烂 SQL/大事务
  3. 主节点 CPU/IO 升配
  4. 仍不够 → PolarDB-X 或业务拆分

8.3 容量

  • 存储水位、临时文件、日志膨胀；
  • 连接数/窗口总和；
  • 活动日按峰值估，不按平均。

10第 9 章 · PolarDB-X 总览：何时上分布式

9.1 上分布式的信号

  • 单主写持续打满且优化空间有限；
  • 数据体量与连接数远超单机舒适区；
  • 业务可接受分片模型（有稳定分片键）。
跨片事务多、全局模糊查多为常态时，分布式可能更慢更贵——先评估工作负载。
应用 → CN（调度）
         ├─ 单片 SQL → 1 个 DN（最快）
         ├─ 多片 SQL → 多 DN 并行 → CN 汇总
         └─ 跨片事务 → 全局协调（最贵）

12第 11 章 · 分片键设计：分布式的生死线

11.1 好键与坏键

类型例子结果
好账号/用户 ID高频交易单片闭环
差纯时间戳最新分片热点
差经常变更的字段重分布痛苦
危险查询无分片条件模糊查广播扫全 DN
设计检查
  1) Top SQL 是否大多单片？
  2) 是否存在必做的跨片事务？比例多少？
  3) 全局唯一如何发号？
  4) 扩容是否可在线、如何校验？
分片键选错，后面所有优化都是打补丁。

13第 12 章 · 分布式事务、全局次序、幂等

跨片转账
  CN 开启全局事务
    → DN_A 预提交
    → DN_B 预提交
    → 全成功则提交，否则回滚
  超时：应用用业务键查终态，禁止盲重试
  • 能单片就单片（性能与稳定性数量级差异）；
  • 全局事务依赖全局提交序/时间戳类机制保证可见性；
  • 金融幂等键建议：业务单号 + 动作类型 + 分片相关键。
「有分布式事务」≠「可以随便跨片设计」。税很重。

14第 13 章 · 全局索引、跨片 Join、分布式 DDL

13.1 全局索引

数据按 A 分、按 B 查时需要。写入要维护索引，成本更高；要评估必要性。

13.2 跨片 Join

尽量下推/改写业务减少大跨片关联；报表可同步到分析库。

13.3 分布式 DDL

变更单审批 → 窗口
  → 元数据编排
  → 各 DN 执行/回填
  → 切换可见
  → 观察慢查询与任务队列
失败 → 按回滚预案，禁止手工改一半

18第 17 章 · 综合案例库（12 个详解）

A. 写后读旧

支付页走 RO。止血：粘主。根治：接口分级规范。

B. RO 被报表打满

摘流、独立地址、优化 SQL、限流。

C. 切换后连接打旧主

连接池重试刷新；查代理拓扑。

D. 长事务堵切换/堵业务

找到来源应用；拆事务；超时门禁。

E. 大事务 DDL 卡死

窗口变更；拆批；回滚预案。

F. PolarDB-X 跨片超时像双扣

先对账；查全局事务终态；幂等。

F 详拆
  超时 → 查全局事务/各 DN
       → 已提交：返回成功并补偿展示
       → 未提交：回滚或人工
       → 应用禁盲重试

G. 分片热点

换键/打散；限流；扩容。

H. 广播查询拖死

加分片条件；全局索引评估；走分析库。

I. 存储水位高

查大表/临时/日志；规范清理；扩容。
