1第 1 章 · 大白话 ↔ 真实流程总对照（全书隐喻地图）

1.1 白话总览

把数据库想成银行网点：办事窗口是你 Java 连接池里借到的那条连接；柜台位子是这条连接上正在跑的 SQL/会话；账本是正在改的那一行数据。窗口不够就排队；位子被长事务占着别人就干不了；账本被人按住（行锁）你就只能等或超时。

1.2 全书隐喻 ↔ 真实组件

大白话真实东西现场你会怎么说挂了/坏了会怎样
办事窗口连接池里的 DB connection池打满、借不到连接、泄漏应用报获取连接超时
柜台位子Session / 正在执行的 SQL活跃会话多、慢 SQL、锁等待业务卡住、超时雪崩
账本（某一行）被锁/被改的行数据热点账号、余额行锁等待、死锁、超时
主柜台RW 读写节点写入口、强一致读入口写失败，需切换
只读柜台RO 只读节点查询分流、报表读慢/摘流；写不受影响
金库底账共享存储（PolarStore 等）真正落盘处极端时整库写不进
金库通道PolarFS 等存储访问层计算访问共享盘的路径IO 抖动、延迟升高
大堂分流集群地址 / 代理 / Proxy读写分离、屏蔽切换连错节点、读写策略错
抄写进度RO 复制/可见性水位（lag）写后读旧的根因结果页短暂不一致
调度大厅PolarDB-X CN收 SQL、改写、下推、汇总入口不可用或计划怪
分柜账本PolarDB-X DN 分片数据切开存放某片慢/不可用
总登记处GMS / 元数据服务拓扑、路由规则、全局信息路由错、DDL 卡住
粘主写后关键读强制走 RW支付结果页策略不做就会「写完读旧」
半成功超时后库侧可能已提交切换/跨片超时现场盲重试可能双扣

1.3 一条支付请求真实走完（对照隐喻）

客户点「确认支付」
  → Java 从连接池借「办事窗口」(connection)
  → 经「大堂分流」(代理) 到「主柜台」(RW)
  → 会话坐上「柜台位子」，改「账本行」(余额/订单状态)
  → 事务提交：改动落到「金库底账」(共享存储)
  → 返回成功；窗口还回池子
结果页若误走「只读柜台」(RO)：
  → 可能「抄写进度」未到 → 页面仍显示未支付（写后读旧）
正确：结果页「粘主」再读，或会话粘滞主柜台

1.4 运维 checklist（每天开口先问这 5 句）

  1. 你们说的 Polar，是共享存储版还是 PolarDB-X？
  2. 写走哪个地址？读是否分流？哪些接口强制粘主？
  3. RO 延迟水位阈值多少？超阈谁摘流？
  4. 连接池最大连接 × 应用实例数，有没有超过库侧上限？
  5. 超时后业务是查单还是直接重试？幂等键在哪？

1.5 常见坑

  • 把 RO 当成「备库私人账本」叙事——共享存储版 RO 是共享金库上的只读计算。
  • 把「有代理」当成「一定不会写后读旧」——代理策略配错一样旧。
  • 开会混用 PolarDB 与 PolarDB-X 术语——CN/DN 只属于分布式那套。

1.6 与 Java 应用侧

窗口=连接：池太小高峰排队；池太大把库会话打满；泄漏=窗口借走不还。超时要分层：连接超时、SQL 超时、业务超时，别三层同一个数字糊弄。

1.7 对照 GaussDB / TDSQL

隐喻通用：主/备或主/只读、连接、会话、行锁。差别在「金库是不是共享」以及「有没有分片调度大厅」。高斯/TDSQL 现场你也用同一套窗口-位子-账本说话，再补各自命令。

先统一语言，再谈架构。隐喻表建议打印贴显示器旁一周。

2.4 运维 checklist

  • 控制台/交付文档里的产品名称与现场口头是否一致；

3第 3 章 · 为什么需要 PolarDB：传统主备痛点拆解

3.1 白话总览

传统主备像「每个柜台自己一本私人账本」，备柜靠抄日志追主柜。PolarDB 共享存储版像「多个柜台共用一座金库」——柜台（计算）可以换，金库（存储）还在。

3.2 传统主备问题（银行语气）

  • 加只读 ≈ 再搬一套盘，耗时长、窗口紧；
  • 备库落后 → 报表/查询读到旧账；
  • CPU 不够常要整机升配，盘和算力绑死；
  • 切换要确认「抄写追平」，RTO/RPO 受复制质量绑架。
传统：应用 → 主库(盘A) ──redo/binlog──▶ 备库(盘B)
问题：盘B 落后=读旧；加备要拷盘；计算与盘绑死；切换看追平

3.3 PolarDB 共享存储怎么解题

PolarDB：应用 → 代理(大堂)
              ├─ RW 主柜台（唯一常规写）
              └─ RO 只读柜台 × N
                    └──── 共享金库（多副本可靠存储）
含义：换柜台 ≠ 搬金库；加 RO 比再搬一套私人账本轻

3.4 失败点（别神话）

失败点现象你先做什么
RW CPU/IO 打满写慢、锁多慢 SQL/热点/升配；别幻想加 RO 能解决写
RO 水位高写后读旧粘主/摘流
共享存储水位或 IO整体抖扩容、查大事务/临时文件
代理策略错读写错乱核对地址与 hint
共享存储版写仍然集中在主计算节点。写到天花板：先治 SQL/热点，再升配，再评估 PolarDB-X 或业务拆分。

3.5 运维 checklist

  • 画出现网：应用实例数、池大小、代理、RW、RO、存储水位；
  • 标出「必须粘主」的接口清单；
  • 标出「只准走 RO」的报表作业，避免打主。

3.6 Java 侧

读写分离要么代理做，要么应用双数据源。双数据源最怕：事务里读走了 RO。规范：同一事务内读也走写库。

3.7 对照

高斯/TDSQL 传统主备痛点类似；差异在复制实现与管控界面。共性：读扩展便宜与否、写是否仍单入口。

4第 4 章 · 共享存储版架构深挖（计算存储分离 / PolarFS·PolarStore 运维视角）

4.1 白话总览

共享存储 PolarDB：多个大脑（计算节点），一座金库（共享存储）。你运维时别只盯 CPU，金库水位和 IO 一样能致命。厂商文档里的 PolarFS / PolarStore，现场可以记成：计算访问金库的「通道与底账系统」——不必背内部论文，但要懂挂了会怎样、监控看什么。

4.2 三层结构（组件 · 数据流 · 状态 · 失败点）

层组件（说法可能因版本而异）职责正常状态失败点与表现
接入集群地址 / 代理 / VIP路由、读写分离、屏蔽切换通、策略符合预期连不上；读到不该读的节点
计算RW、ROSQL 解析、优化、事务、锁、缓冲池RW 可写；RO 可读且水位可控RW 宕→切换；RO 宕→摘流
存储访问PolarFS 等（运维视角：访问层）把页/日志读写落到共享存储延迟稳定IO 毛刺、抖动、超时
共享存储PolarStore 等（运维视角：底账）数据页与日志持久化、多副本水位健康、副本齐全空间满、写失败、极端只读
管控云管控 / 本地管控平面探活、切换编排、备份任务任务成功、事件可查切换异常、备份失败
写路径（逐步）
1. 应用借连接（办事窗口）→ 打到代理/主地址
2. RW 鉴权、解析 SQL、生成计划
3. 在缓冲里改数据页；按隔离级别加锁（占账本）
4. 产生 redo/事务日志，经存储访问层持久化到共享存储（落定）
5. 返回客户端成功
6. RO 通过共享存储+同步/可见性机制看到新版本（有水位）

读路径
SELECT → 代理判定主或 RO
      → 计算节点缓冲命中？命中则回
      → 否则从共享存储拉页 → 返回

4.3 PolarFS / PolarStore 运维该懂的「程度」

  • 不必：把论文级实现背下来应付日常值班。
  • 必须：知道计算挂了数据还在；存储满了谁都救不了；IO 抖动会让「没事的 SQL」集体变慢。
  • 巡检：存储空间水位、IOPS/吞吐/延迟、副本健康（以控制台指标名为准）、备份依赖的快照/日志链是否完整。

4.4 状态机（简化）

RW:  Running ──探活失败──▶ 切换中 ──成功──▶ 新 RW Running
                              └──失败/人工──▶ 应急只读/止损
RO:   InService ──水位高──▶ 摘流观察 ──恢复──▶ InService
Storage: OK ──水位>阈──▶ 扩容窗口 ──满──▶ 写拒绝（事故）

4.5 现场运维 checklist

  1. 拓扑：RW 1 + RO N，规格是否对称或按读写配比；
  2. 连接：应用是否全走集群地址；
  3. 存储：水位、近 7 日增长、大表/临时/日志膨胀；
  4. IO：是否与慢 SQL、大事务、批量作业同峰；
  5. 事件：近 7 日有无切换、只读摘流、备份失败。

4.6 坑 + 银行案例

案例：日终批量把 RW CPU 打满，有人建议「再加两个 RO」。RO 加了写压力纹丝不动。根因：写在主柜台。处置：拆批、错峰、升配 RW、治理大事务；RO 只救读。

4.7 Java 侧

连接串指向集群地址；不要缓存「旧主」IP。连接池在切换后要能重连（检测查询 + 合理超时）。批量作业单独账号与限流，避免与联机抢同一池。

5第 5 章 · 一写多读、RO 水位与读写分离策略（写后读旧专题）

5.1 白话总览

主柜台改完账，只读柜台可能还差几秒「抄写进度」。支付成功页若走只读柜台，客户会看到「没付上」——钱其实已扣，是读旧。这叫写后读旧，银行联机里非常典型。

5.2 原理：水位与可见性

RO 不是另一套完全独立私人账本（共享存储语境），但仍有可见性/复制水位。水位高 = 读到的版本偏旧。代理若按 SQL 类型自动把 SELECT 甩到 RO，结果页中招。

写成功？
  ├─ 强一致接口（支付结果/余额/冲正前核对）→ 必须粘主 / 读 RW
  ├─ 可延迟（浏览、列表、内容）→ 可读 RO，盯水位
  ├─ 报表/导出 → RO 或独立只读地址，限流隔离
  └─ 水位超阈 → 摘流 RO 或提示稍后；关键读回落主（注意主压）

5.3 接口分级表（可直接贴团队墙）

5.4 代理策略 checklist

  • 是否按 SQL 自动分流？事务中 SELECT 是否仍可能进 RO？

  • 延迟阈值多少？谁告警、谁摘流、谁恢复？
  • 只读地址是否物理隔离给报表账号？

5.5 银行案例

6.1 白话总览

  → 存储侧认新写者租约（共享存储语境）

8第 8 章 · 备份、时间点恢复（PITR）、误删应急

8.1 白话总览

备份不是「控制台有个绿勾就安心」。银行要的是：删错表、错发 UPDATE、错误变更后，能回到时间点 T，且先在临时实例验证，再导回生产——不能直接盖生产碰运气。

8.2 原理（运维视角）

共享存储语境下，常见是快照/数据页 + 日志链，支持按时间点恢复（以你们版本与交付为准）。没有日志链，PITR 就是空话。

定期快照/基备 ─┐
连续日志保留  ─┼→ 选时间点 T → 恢复到「临时实例」
手工全备      ─┘         → 校验数据 → 导出缺失/订正 → 回灌生产
                         → 复盘权限与变更门禁
禁止：不经验证直接覆盖生产

8.3 RPO / RTO

指标含义现场怎么谈
RPO最多丢多久数据日志保留与备份频率决定上限
RTO多久恢复可服务恢复演练测出来的，不是幻灯片

8.4 误删表标准动作

  1. 立刻停继续破坏的操作（禁继续删、禁再跑错脚本）；
  2. 确认大约时间点与影响对象；
  3. 恢复到临时实例，不对生产下手；
  4. 导出缺失数据，按主键/业务键回灌；
  5. 对账；
  6. 复盘：谁的权限、为何能在生产直接 DDL/大 UPDATE。

8.5 运维 checklist（日/周）

  • 备份任务成功？失败是否告警到人？
  • 保留天数是否满足监管/行内制度？
  • 最近一次恢复演练记录：时间、RTO、问题单；
  • 日志空间是否因大事务异常膨胀。

8.6 银行案例

案例：在生产把测试 WHERE 条件跑丢，更新了整表状态位。止血：停应用写；按备份链恢复临时库对比；用业务时间窗订正。根治：生产变更强制 WHERE 主键预览行数 + 双人复核。

11第 11 章 · PolarDB-X 总览：何时上分布式、何时不上

11.1 白话总览

PolarDB-X 像把金库拆成很多「分柜账本」，前面加「调度大厅」(CN) 接待。单柜办得完的事很快；要跑多个分柜一起办（跨片事务、跨片 Join），手续费很贵。上分布式是因为单主写/体量顶不住，不是因为「听起来更高级」。

11.2 上分布式的信号

  • 单主写持续打满，SQL/热点优化空间有限；
  • 数据体量、连接数远超单机舒适区；
  • 业务有稳定分片键（账号、客户号等），Top SQL 大多可单片闭环；
  • 组织能接受：全局模糊查变贵、跨片事务变少变严、DDL 要编排。

11.3 明确不要轻易上的信号

  • 跨片事务是常态（例如任意两账户转账无路由策略）；
  • 大量无分片条件的模糊查询、复杂报表在线跑；
  • 分片键经常变更；
  • 团队不会做幂等与对账。
应用 → CN（调度大厅）
         ├─ 单片 SQL → 1 个 DN（最快，目标态）
         ├─ 多片 SQL → 多 DN 并行 → CN 汇总（贵）
         └─ 跨片事务 → 全局协调（最贵，金融要慎）

11.4 运维 checklist（立项评审）

  1. Top 20 SQL 是否单片？比例？
  2. 跨片事务类型与日均量？
  3. 全局唯一号怎么发？
  4. 全局索引需求？写入放大可接受吗？
  5. 扩容是否可在线、校验怎么做？

11.5 银行案例

案例：为「以后更大」提前上 X，结果 80% 接口跨片，延迟与超时上升。回退成本极高。教训：先用数据证明单机写墙，再切。

11.6 Java 侧

分布式下超时更常见，幂等与查单成为标配。连接仍打 CN 入口，不要直连某 DN（除非厂商运维通道）。

14第 14 章 · 分布式事务、全局次序、幂等与对账

14.1 白话总览

跨片转账=多个分柜必须一起成或一起回滚。大厅要做全局协调，手续慢、超时多。金融现场：能单片就单片；必须跨片就配幂等与对账，禁止盲重试。

跨片转账（示意）
  CN 开启全局事务
    → DN_A 预提交（预留出账）
    → DN_B 预提交（预留入账）
    → 全成功则提交，否则回滚
  客户端超时：
    → 用业务键查终态（成功/失败/处理中）
    → 禁止直接再扣一遍

14.2 原理要点

  • 单片本地事务：便宜、稳，目标态；
  • 跨片全局事务：依赖全局提交序/时间戳类机制保证可见性（实现因版本而异）；
  • 超时不等于失败，也不等于成功——是「未知」，要查单；
  • 幂等键建议：业务单号 + 动作类型 + 分片相关键。
模式适用优点风险
单片事务本账户操作快模型要配合
全局事务强一致跨片语义清晰延迟、超时、锁面
本地消息表+重试可最终一致稳、好对账要补偿设计
TCC高并发长流程灵活代码复杂

14.3 运维 checklist

  • 全局事务成功率、耗时 P99；
  • 悬而未决事务是否有巡检；
  • 对账任务是否覆盖跨片通道。

14.4 银行案例

案例：跨片超时，网关自动重试 3 次，疑似双入账。止血：停自动重试；按业务单号对账；根治：幂等唯一约束 + 超时只查单。

「有分布式事务」≠「可以随便跨片设计」。税很重。

15第 15 章 · 全局索引、跨片 Join、分布式 DDL 风险

15.1 白话总览

数据按账号分片，却要按证件号查——就可能上全局索引。写入要多维护一份「目录」，成本更高。跨片 Join 像把多个分柜的账本搬到大厅对表，报表一跑联机发抖。分布式 DDL 是搬家工程，不是「随便 ALTER 一下」。

15.2 全局索引

问题说明
何时需要高频按非分片键点查，且不能改模型
代价写入放大、存储增加、索引分片自身热点
替代映射表、搜索平台、客户号反查服务
运维看什么索引延迟、维护失败、写入 RT 上升

15.3 跨片 Join

跨片 Join 成本
  多 DN 拉数据 → CN 做关联/排序
  → 网络与内存暴涨 → 易超时
对策：
  业务改写下推；冗余字段避免 Join；
  报表走分析库/宽表；严禁大范围在线跨片

15.4 分布式 DDL

变更单审批 → 窗口
  → 元数据编排
  → 各 DN 执行 / 回填 / 校验
  → 切换可见
  → 观察慢查询、任务队列、错误率
失败 → 按回滚预案，禁止手工改一半
  • 加字段/加索引：评估锁与回填时间；
  • 改分片规则：等同搬家，需演练；
  • 大表 DDL：错峰、限速、双人。

15.5 银行案例

案例：业务高峰对大表加全局索引，写入 RT 翻倍，渠道报警。止血：停 DDL/限速；根治：窗口 + 灰度 + 回滚点。

15.6 Java 侧

ORM 懒加载跨对象关联极易变跨片。领域模型按分片边界切，别把「一个聚合」设计成跨很多片的隐式 Join。

  4. 回滚步骤（谁执行、多久内有效）；
  5. 观察指标：错误率、延迟、锁、水位、空间、慢 SQL；

23第 23 章 · FAQ 长章（现场高频问）

Q1. PolarDB 和 PolarDB-X 到底啥区别？

前者经典形态是共享存储一写多读；后者是分片分布式。开会第一句先确认形态。

Q2. 有 RO 是不是就强一致了？

不是。关键读要粘主或可接受延迟。

Q3. 加 RO 能解决写慢吗？

不能。写在 RW。加 RO 只扩读。

Q4. 写后读旧一定是数据库bug吗？

多半是策略问题：读到了水位未到的 RO。

Q5. 切换成功为什么业务还错？

半成功+盲重试。要查单对账。

Q6. 连接池要开多大？

先算库连接预算，再倒推实例×池；不是越大越好。

Q7. 超时了重试行不行？

资金类：先查单，有幂等才能重试。

Q8. 长事务怎么快速定位？

查会话/事务视图→SQL→来源应用→Java 事务边界。

Q9. 存储到多少该扩？

建议 70% 预警、85% 行动，按日增外推。

Q10. 备份绿勾就万事大吉？

没有恢复演练等于没备份。

Q11. 误删表先做什么？

停破坏→确认时间点→临时实例恢复→回灌→复盘权限。

Q12. 啥时候上 PolarDB-X？

单主写墙+有稳定分片键+跨片可控。否则慎上。

Q13. 分片键用日期行吗？

单独用很容易热点；日期更适合分区/归档。

Q14. 全局索引是银弹吗？

不是。写入放大；先评估替代方案。

Q15. 跨片 Join 报表能在线跑吗？

高峰尽量别；走分析库/宽表。

Q16. 分布式 DDL 失败怎么办？

按预案回滚，禁止手工改一半。

Q17. CN 挂了和 DN 挂了咋区分？

全员进不去偏 CN；部分号段异常偏 DN。

Q18. 和 GaussDB 比 Polar 优势？

现场别吵优劣；抓共性链路，补各自手感。

Q19. 和 TDSQL 分片比？

决策题同构：键、跨片税、DDL。

Q20. 代理回落主危险吗？

RO 全挂回落可能压死主，要熔断比例。

Q21. 如何防双扣？

幂等键+唯一约束+超时查单+禁默认重试。

Q22. 日终和联机抢资源怎么办？

通道隔离：账号、地址、限流、错峰。

Q23. 参数改完不生效？

区分是否重启类；变更单写清。

Q24. 只读账号能在主库跑报表吗？

能跑但不该；会抢写资源。

Q25. 如何证明容量够？

用活动日 P95，不是平均值。

Q26. Java 事务里能读 RO 吗？

不建议；同一事务走写库。

Q27. 粘主要粘多久？

至少覆盖结果查询窗口；可用会话标记。

Q28. 监控指标英文名记不住？

记大白话含义+控制台位置即可。

Q29. 厂商说 RPO=0 还要演练吗？

要。演练测的是人和流程的 RTO。

Q30. 双写迁移最怕什么？

一侧失败不对账就切流。

Q31. 热点账号怎么办？

排队、合并更新、限流；分布式也要防热点片。

Q32. 死锁一定是数据库问题？

多半是加锁顺序不一致。

Q33. 应用扩容后库扛不住？

连接数线性涨，要同步重算池与库配额。

Q34. PolarFS 我要学多深？

运维懂通道与失败表现即可，不必先啃论文。

Q35. 如何带组员入门？

先隐喻表，再形态矩阵，再五步法案例。

Q36. 笔记丢了怎么办？

定期导出 JSON；换电脑先导入。
